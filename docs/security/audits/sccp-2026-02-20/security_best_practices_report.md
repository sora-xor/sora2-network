# SCCP Security Best Practices Report

Date: 2026-02-24 (updated)

## Executive Summary
This audit reviewed SCCP runtime logic (`pallets/sccp`), SCCP MCP tooling (`misc/sccp-mcp`), and SCCP runtime integration tests. No direct unauthorized mint/burn bypass was identified in the reviewed code paths. Initial high/medium findings have now been remediated, with evidence captured below. Additional MCP availability hardening now includes explicit RPC timeout controls, bounded retry controls for retry-safe read paths, circuit-breaker fail-fast behavior during sustained upstream failure, bounded in-flight RPC load-shedding caps at global, per-endpoint, per-principal, per-tool-scope, and per-method scopes, and an opt-in weighted per-principal admission queue with bounded pending caps/timeouts. Follow-up hardening now also includes direct malformed-input/fuzz-style coverage for EVM/TRON proof helper modules, dedicated `cargo-fuzz` harnesses for those helper modules, constant-time MCP auth token comparison, bounded auth token length policy (minimum + maximum), bounded per-principal auth-failure backoff with tracked-principal state limits, submit-tool security audit logging, and a dedicated SCCP activation preflight checker for domain/token misconfiguration risk.

## High Findings

### SCCP-001: Placeholder Runtime Weights For Proof-Heavy Paths (Status: Remediated)
Impact: Underestimated execution costs on proof/header verification paths can enable block-level DoS and reduce liveness.

Evidence:
- `runtime/src/lib.rs:2367` (`type WeightInfo = sccp::weights::SubstrateWeight<Runtime>;`)
- `runtime/src/lib.rs:3556` / `runtime/src/lib.rs:3655` (SCCP benchmark registration)
- `pallets/sccp/src/benchmarking.rs:1` (benchmark module added)
- `pallets/sccp/src/weights.rs:1` (autogenerated benchmark-based weights)
- `pallets/sccp/src/weights.rs:201` / `pallets/sccp/src/weights.rs:227` / `pallets/sccp/src/weights.rs:461` (conservative floors for under-exercised setup paths)
- `.github/workflows/static_analysis.yml:45` (CI enforcement step)
- `.github/scripts/check_sccp_runtime_weightinfo.sh:16` (regression guard script)

Why this matters:
- Proof/header handling is variable-cost work.
- SCCP previously used fixed placeholder weights in runtime.
- Even with proof bounds, decode/verification work can still vary materially by input and verification mode.

Implemented remediation:
1. Replaced unit `WeightInfo` wiring with benchmarked SCCP weights in runtime.
2. Added SCCP pallet benchmarks and generated `pallets/sccp/src/weights.rs`.
3. Added CI guard to fail if SCCP runtime wiring regresses from `SubstrateWeight<Runtime>`.
4. Applied conservative weight floors for benchmark paths that currently rely on lightweight/synthetic setup to avoid underweighting.

### SCCP-003: No Built-In Authentication Layer In MCP Tool Call Path (Status: Remediated)
Impact: With internet exposure, unauthenticated clients can execute allowed MCP tools.

Evidence:
- `misc/sccp-mcp/src/config.rs:284` (`resolve_auth_token_for_startup` resolves mandatory auth token)
- `misc/sccp-mcp/src/config.rs:290` (`validate_startup_policy` fail-closed checks)

- `.github/workflows/static_analysis.yml:50` (CI enforcement step)
- `docs/security/sccp_mcp_deployment_guardrails.md:1` (required deployment controls)

Why this matters:
- MCP server is stdio-oriented and trusts whoever can invoke `tools/call`.
- This is safe for local/isolated use, but not safe as-is for internet-facing exposure.

Implemented remediation:
1. Keep MCP behind an authenticated gateway (mTLS/OIDC/service-auth).
2. Restrict who can reach the stdio bridge process.
3. Added operator guardrails and fail-closed startup checks for risky exposure patterns unless explicitly acknowledged.
4. Added mandatory built-in request auth token verification for `tools/list` and `tools/call`.
5. Added startup token resolution (inline or env-based) and fail-closed refusal when no token is configured.
